# ASD 干预与评估助手 — 第一版设计文档

> 子项目隶属:南京特殊教育师范学院 · 特殊教育大模型研发项目
> 文档日期:2026-06-18
> 状态:设计已确认,待用户复核

## 1. 背景与定位

学校面向全国 2300 多所一线特殊教育学校、康复机构,以特殊教育智能化为宗旨建设特殊教育领域大模型。本子项目是该工程拆解后的**第一个落地子项目**,聚焦"干预与评估"方向。

整个特教大模型工程拆解为五个独立子项目,各自走"设计→计划→实现"循环:

0. 数据与知识底座(知识库 + RAG + 合规框架)— 地基,可与第一个应用并行起步
1. **干预与评估助手 ← 本文档,第一优先**
2. 教学助手(备课、个别化教案、资源)
3. 科研助手(文献、量表统计、写作)
4. 智能教具(AR/多模态/具身)— 后期或与厂商共建

选择"干预与评估助手"作为第一个子项目的理由:痛点最尖锐(专业人员奇缺)、数据闭环最强、最能立住高校的专业壁垒。

## 2. 第一版范围(已确认)

| 维度 | 决策 |
|---|---|
| 核心链路 | 评估 → 报告 → IEP 草案,完整轻闭环(每步做轻,先跑通) |
| 障碍类型 | 聚焦孤独症谱系障碍(ASD) |
| 角色 | 三角色同步:老师/康复师、管理者、家长 |
| 技术形态 | Web(老师/管理者端)+ 微信小程序(家长端) |
| 评估引擎 | 通用量表引擎 + 内置 1-2 个量表样例 |
| AI 能力 | 外部大模型 API + RAG;模型调用层可插拔,后续可切私有化/微调 |
| 数据合规 | 脱敏 + 合规企业版 API 双保险(强制) |
| 成功标准 | 3-5 所样板校真实跑通闭环 + 功能完整度 |
| 实现路径 | 方案一:模块化单体(务实快交付) |

### 不在第一版范围(YAGNI)

- 其他障碍类型(智力障碍、言语障碍等)的专属量表与干预范式
- 模型私有化部署与领域微调(架构预留切换能力,但不在第一版实现)
- 智能教具/AR、科研助手、教学助手等其他子项目
- 家长端的复杂交互(第一版家长端仅轻查看 + 居家建议 + 反馈)

## 3. 关键设计红线

1. **数据出网必经脱敏层**:任何发往外部大模型 API 的数据,身份信息(姓名、学校、身份证等)必须先在本地替换为占位符;脱敏校验不通过则硬阻断,绝不发送。
2. **AI 只产草案,人来定稿**:评估结论、报告、IEP 均由 AI 生成草稿/草案,最终必须由专业人员(老师/康复师)审阅定稿。AI 不直接对儿童下结论。
3. **敏感数据自主可控**:服务对象为残障儿童(未成年人 + 敏感个人信息双重高敏),原始身份信息加密留存本地,永不出网。

## 4. 系统架构(方案一:模块化单体)

分三层:

### ① 客户端层
- **Web 管理端**:Vue 3 + Vite 5 + Element Plus + Pinia + Vue Router;服务老师/康复师与管理者,承载评估、报告、IEP、管理等重交互。
- **微信小程序(家长端)**:uni-app(Vue3 + Pinia)编译微信端;家长查看 IEP/报告、接收居家干预建议、提交反馈、查看进度趋势。

### ② 后端单体(Spring Boot 3,模块化分包)
- 业务模块:user、child、assessment、report、iep、feedback
- 共享能力层:anonymizer(脱敏)、ai-gateway(统一 LLM 接口)、rag(检索)、common(统一返回/异常/审计)

### ③ 数据与外部服务
- MySQL 8(业务数据)、Redis 7(缓存/会话)、MinIO(报告/附件)、向量库(RAG 知识,如 Milvus 或 PgVector)
- 外部大模型 API(合规企业版,仅接收脱敏数据)

## 5. 核心数据流:评估 → 报告 → IEP 草案

1. **评估**(老师/康复师 · Web):选量表 → 录入观察/作答 → 量表引擎自动计分 → 落库(原始身份信息留本地 MySQL,加密)。
2. **🔒 脱敏层**(强制):姓名/学校/身份证 → 占位符;只把"分数 + 必要行为描述"打包出网。
3. **报告生成**(ai-gateway + rag):RAG 召回 ASD 解读规则/常模 → 大模型生成评估报告草稿 → 返回 → 本地还原身份信息后展示。
4. **老师审阅修改**:人把关,确认报告。
5. **IEP 草案**(ai-gateway + rag,再次经脱敏层出网):RAG 召回 IEP 范例/干预策略库 → 生成长短期目标 + 干预活动建议草案。
6. **老师编辑定稿** → 版本保存 → 家长端(小程序)可查看 IEP/报告、收居家建议、反馈。
7. **↺ 数据回流**:老师对 AI 草案的修改 = 高质量训练信号,脱敏后沉淀为专属语料(数据飞轮)。

## 6. 数据模型

### 基础
- **User**:id、角色(老师/管理者/家长)、所属机构
- **Organization**:id、名称、区域
- **Child**(🔒):id、身份信息(本地加密)、障碍类型、关联家长/老师

### 量表引擎(通用,三件套)
- **Scale**:量表定义(id、名称如 CARS、版本)
- **ScaleItem**:题目(题干、选项、维度)
- **ScoringRule**:计分规则(分段、常模、解读阈值)

### 业务记录
- **Assessment**:评估记录(关联 Child + Scale、作答、得分、评估人、时间)
- **Report**:报告(关联 Assessment、AI 草稿、定稿、状态)
- **IEP**:计划(关联 Child、长期目标)
- **IEPGoal**:目标(长/短期目标、衡量标准、状态)
- **IEPVersion**:版本(每次定稿存一版,可追溯)
- **ProgressRecord**:康复进度记录(关联 Child + IEPGoal、达成度、表现、用时、记录人、日期)
- **ProgressTrend**:进度趋势视图(聚合同一目标的多条 ProgressRecord + 跨时间 Assessment 得分 → 趋势曲线)

### AI 与回流
- **AIGenerationLog**:提示/模型/版本、AI 原稿 vs 人工终稿(回流语料)
- **KnowledgeDoc**:RAG 文档、向量、来源
- **Feedback**:家长/老师对 IEP/报告的反馈

### 数据模型要点
- Child 身份字段本地加密,与脱敏解耦。
- 量表拆成"定义/题目/计分规则"三件套,支撑通用引擎。
- 进度记录挂在 **IEPGoal(短期目标)** 上,而非泛泛挂在孩子身上 —— 进步可精确归因到具体干预目标,趋势才有意义。
- AIGenerationLog 同时存 AI 原稿与人工终稿,天然形成回流语料。
- 关系链:Child → IEP → IEPGoal → 多条 ProgressRecord → 聚合成 ProgressTrend。

## 7. 模块划分与接口边界

### 业务模块(单一职责,通过 Service 接口 + DTO 通信)
| 模块 | 职责 |
|---|---|
| user | 认证、三角色 RBAC、机构 |
| child | 儿童档案、身份加密 |
| assessment | 量表引擎、计分 |
| report | 报告草稿、定稿 |
| iep | 目标、版本、进度 |
| feedback | 家长/老师反馈 |

### 共享能力层(接口稳定,实现可替换)
| 能力 | 职责 |
|---|---|
| 🔒 anonymizer | 脱敏/还原,出网必经 |
| ai-gateway | 统一 LLM 接口,实现可从 API 切换到私有化/微调 |
| rag | 检索召回、知识库管理 |
| common | 统一返回、异常、审计日志 |

### 关键约束
1. 业务模块**不直接**调外部 API,一律走 ai-gateway。
2. ai-gateway 调模型**前必经** anonymizer;业务模块拿到的是还原后数据。
3. report/iep 依赖 ai-gateway + rag,但**不感知**背后是哪个模型(可插拔)。
4. 模块间只通过 Service 接口 + DTO 通信,不直接碰对方的表。

> 这套边界让"换模型/加量表/拆微服务"都只动局部:换模型只动 ai-gateway 实现;拆服务时模块已是天然边界。

## 8. 错误处理

| 环节 | 策略 |
|---|---|
| 🔒 脱敏失败 | **硬阻断**:校验不通过则绝不发送外部 API,直接报错。宁可不出结果,不可泄露身份信息。 |
| AI 调用失败/超时 | **优雅降级**:重试 + 超时控制;最终失败则保留已计分的评估数据,提示"AI 草稿生成失败,可重试或手动撰写",不影响评估与计分主流程。 |
| 计分异常 | **校验前置**:作答完整性、量表版本匹配在计分前校验;计分规则缺失则拒绝出分并提示,杜绝错误分数流入报告。 |
| 统一处理 | common 模块统一返回结构与异常码;所有 AI 调用、脱敏、数据访问留审计日志(合规可追溯)。 |

## 9. 测试策略

- **单元测试(重点)**:量表计分规则(各分段/边界值)、脱敏层(身份信息必须被替换、能正确还原)。这两块是正确性与合规的命门,必须高覆盖。
- **集成测试**:评估→报告→IEP 主链路打通;ai-gateway 用 **Mock 模型**测试,不依赖真实 API,保证可重复、低成本。
- **AI 输出验证**:AI 是非确定性的,不做精确断言;验证"结构合规"(字段齐全、格式正确)+ 关键约束(输出不含被脱敏的身份信息)。质量靠样板校人工评估闭环把关。

## 10. 资源需求(由方案倒推)

方案一为模块化单体,团队需求相对可控。粗略倒推(供立项参考,非承诺工期):

- **后端**:1-2 人,Spring Boot 3 + MyBatis + MySQL/Redis/MinIO,负责业务模块、量表引擎、脱敏层、ai-gateway、rag 接入。
- **前端**:1 人,Vue 3 管理端;家长端 uni-app 小程序工作量可由同一人分期承担或增配 0.5 人。
- **算法/AI**:0.5-1 人,负责 RAG 知识库构建、提示工程、AI 输出质量评估;第一版不训模型,投入偏集成与调优。
- **特教专业支持**:学校侧专家投入(量表选型、计分规则、IEP 范例、报告解读规则、样板校对接)—— 这是专业壁垒所在,非工程人力可替代。
- **合规/伦理**:法务与伦理委员会前置介入,建立数据采集、脱敏、知情同意、存储的合规框架。

> 建议分阶段交付:先 2-3 个月出核心闭环(单量表打通评估→报告→IEP + 脱敏 + Web 老师端),再迭代补全三角色、小程序家长端、通用量表引擎、进度追踪,配合样板校验证与项目申报节点。
